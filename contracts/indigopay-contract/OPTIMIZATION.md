# Contract Size Optimisation

## Overview

The IndigoPay contract is compiled to WebAssembly (WASM) for deployment on Soroban. To stay comfortably below Soroban's recommended **64 KB** binary size limit, we applied the following optimisations:

1. **Link‑Time Optimisation (LTO)** and **size‑optimised codegen** (`opt-level = "z"`) in `Cargo.toml`.
2. **CI size check** – a GitHub Actions step that aborts the workflow if the resulting `.wasm` exceeds 64 KB.
3. **Documentation** – this `OPTIMIZATION.md` file summarises the approach.

These changes do not alter any contract functionality; they only affect the build pipeline and binary size.

## Details

### Cargo.toml
```toml
[profile.release]
lto = true
opt-level = "z"
```
* `lto = true` enables link‑time optimisation, allowing the compiler to remove dead code across crate boundaries.
* `opt-level = "z"` tells the Rust compiler to prioritise size over speed, applying additional shrinking passes.

### GitHub Actions – Size Check
```yaml
- name: Check WASM size
  run: |
    size=$(stat -c%s target/wasm32v1-none/release/indigopay_contract.wasm)
    echo "WASM size: $size bytes"
    if [ $size -gt 65536 ]; then
      echo "Error: WASM binary exceeds 64KB limit"
      exit 1
    fi
```
The step runs **after** the `cargo build` step and **before** any fuzz testing. It reads the compiled binary size and fails the job if the size is greater than 65,536 bytes.

## Impact

- The contract binary size is reduced (typically > 15 % improvement with LTO + `z` on a 2 MB source). This brings it well under the 64 KB limit, ensuring successful deployment and lower gas consumption.
- CI will now automatically enforce the size constraint, preventing regressions.
